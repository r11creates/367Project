--------------------------------------------------------------------------------
1. PROBLEM AND GOAL
--------------------------------------------------------------------------------
The starter code computes forces between all pairs of particles (O(n^2)). The
assignment requires:
  - A serial implementation that runs in O(n) time (linear in number of particles).
  - An OpenMP implementation that parallelizes this O(n) solution for shared memory.

The key fact from the handout: with the given low density, we expect only O(n)
interactions total. So we must avoid checking every pair and instead only consider
particles that can actually be within the cutoff distance.

--------------------------------------------------------------------------------
2. WHY SPATIAL BINNING (CELL LIST)
--------------------------------------------------------------------------------
Forces only act when two particles are closer than CUTOFF (0.01 in common.cpp).
So for each particle we only need to look at "neighbors" within a small distance.
If we partition the simulation box into cells whose side length is at least
CUTOFF, then any particle within CUTOFF of a given particle must lie in the same
cell or in one of the 8 adjacent cells (3x3 block in 2D). So we:
  1. Assign each particle to a cell based on its (x,y).
  2. For each particle, only loop over particles in the same cell and the 8
     neighboring cells.

That way we do O(1) work per particle on average (constant expected neighbors
per cell at low density), so total force computation is O(n). The cost of
building the cell list each step is also O(n), so the whole step is O(n).

--------------------------------------------------------------------------------
3. SERIAL IMPLEMENTATION (serial.cpp)
--------------------------------------------------------------------------------
Thought process:
  - We cannot change common.cpp/common.h, so we duplicate the constants CUTOFF
    and DENSITY in our file (same values as common.cpp) to compute the box size
    and cell grid. Box size = sqrt(DENSITY * n), matching set_size() in common.
  - Number of cells: we want cell width >= CUTOFF so that the 3x3 neighborhood
    is sufficient. So numberCellsX = (int)(simulationSize / CUTOFF), and same
    for Y. If that is 0 we use 1. Cell width is then simulationSize / numberCellsX,
    which is >= CUTOFF.
  - Data structures: cellFirst[totalCells] = index of first particle in that cell
    (-1 if empty). nextInCell[n] = for each particle, the next particle index in
    the same cell (-1 if none). So each cell is a linked list of particle indices.
  - Each time step:
      a) Clear: set all cellFirst[c] = -1.
      b) Bin: for each particle, compute its cell (cellX, cellY) from (x,y),
         clamp to valid range, then add the particle to that cell's list by
         nextInCell[i] = cellFirst[cell], cellFirst[cell] = i.
      c) Forces: for each particle i, get its cell (centerCellX, centerCellY).
         Loop over the 3x3 block of cells (offsetX, offsetY in {-1,0,1}). For
         each cell in the block, traverse the linked list (j = cellFirst[cellIndex],
         then j = nextInCell[j] until -1) and call apply_force(particles[i],
         particles[j], ...) for each neighbor j.
      d) Move: unchanged, for each particle call move().

Why this is correct:
  - Any pair within distance CUTOFF lies in the same or adjacent cells, so we
    never miss an interaction. We might check a few extra pairs (in adjacent cells
    but just beyond CUTOFF); apply_force returns immediately when r2 > cutoff^2,
    so that is harmless. So physics is unchanged and we get O(n) time.

--------------------------------------------------------------------------------
4. OPENMP IMPLEMENTATION (openmp.cpp)
--------------------------------------------------------------------------------
Thought process:
  - Use the same cell list and constants as serial so that the algorithm remains
    O(n) and correct.
  - Parallelize only where it is safe and beneficial:
      a) Binning: If we let multiple threads push into the same cell list, we
         would need locks (e.g. omp_lock_t per cell) and contention could hurt.
         So we do binning in a single thread (#pragma omp single). It is O(n)
         and typically a small fraction of the step; the rest is force computation.
      b) Force computation: Each particle i is written only to particles[i].ax
         and particles[i].ay. So we can parallelize over particles with
         #pragma omp for. Each thread needs its own dmin for the correctness
         stats; we use private(dmin) and after the loop we use #pragma omp critical
         to update the global absmin. For davg and navg we use reduction(+:davg,navg)
         so OpenMP combines per-thread values automatically.
      c) Move: Each thread updates different particles; #pragma omp for is safe.
  - We use schedule(static) on the force and move loops so each thread gets a
    contiguous chunk of particle indices. That improves cache locality (each
    thread touches a contiguous region of the particles array) and can reduce
    false sharing.
  - Barrier: After the single region (binning) we need all threads to see the
    updated cell lists before the force loop, so we use #pragma omp barrier.

Synchronization and locking:
  - Only one critical section per step: updating absmin from per-thread dmin. The
    rest uses OpenMP reductions (davg, navg) and no locks. Binning is serial so
    no locks there. This keeps synchronization overhead low.

--------------------------------------------------------------------------------
5. DESIGN CHOICES SUMMARY
--------------------------------------------------------------------------------
  - Cell width >= CUTOFF: ensures 3x3 neighbor search is sufficient for correctness.
  - Linked-list cell representation: simple, no dynamic allocation per step, O(1)
    insert per particle when binning.
  - Serial binning in OpenMP: avoids per-cell locks and contention; binning is
    O(n) and small compared to force work for large n.
  - Static scheduling: contiguous chunks per thread for better cache behavior.
  - Single critical for dmin: minimal locking; reductions handle davg and navg.

--------------------------------------------------------------------------------
6. CORRECTNESS CHECK (from handout)
--------------------------------------------------------------------------------
A correct simulation gives:
  - absmin (minimum distance between any two particles during the run) > 0.4,
    typically in 0.7-0.8.
  - absavg (average distance) ~0.95 when most particles interact correctly.

An incorrect simulation (e.g. missing interactions) gives absmin in 0.01-0.05
and absavg ~0.66. Our runs show absmin in the 0.73-0.79 range and absavg ~0.95,
so the implementation is correct.

--------------------------------------------------------------------------------
7. PERFORMANCE NOTES
--------------------------------------------------------------------------------
  - Serial: Autograder fits log(time) vs log(n) and reports a slope. Slope near
    1.0 means O(n). Our slopes are around 1.03-1.09 (slightly above 1, e.g. cache
    effects or constant overhead).
  - OpenMP: Strong scaling = same n, more threads; weak scaling = n grows with
    threads so work per thread is roughly constant. Efficiency and speedup are
    reported by the autograder. On Scinet with Intel compiler and more cores,
    results may differ from local g++ runs; always use Scinet for final numbers
    and the report.
